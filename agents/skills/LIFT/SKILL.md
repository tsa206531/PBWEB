# 產品官網設計框架（LIFT × 信任）

## 核心哲學

官網只有一個工作：讓對的人在幾秒內「看懂 → 相信 → 行動」。

建站初期的錯誤，幾乎都源自同一件事——把「開發者視角」（版本號、功能清單、內部術語）誤當成「訪客視角」（我有什麼痛？你怎麼解？我為什麼要信你？）。這份 skill 用 LIFT 模型當尺，逐項檢查網站有沒有踩雷。

**LIFT 六因子**（WiderFunnel）：價值主張是核心，其餘五個是槓桿——相關性、清晰、急迫性會「拉高」轉換；焦慮、分心會「拖累」轉換。

**權重隨產品類型改變（重要）：**
- 一般 SaaS／消費品 → 六因子平均使力。
- **信任型產品**（資安、隱私、金融、健康、開發者工具、任何處理敏感資料的 App）→ **焦慮與清晰權重最高**；急迫性幾乎放掉，因為假急迫會直接砸信任。
- 免費／開源工具 → 急迫性弱，改用「開源、local-first、可稽核」等信任訊號當主要說服力。

## 使用方式（三步）

1. 用「一、LIFT 逐項檢查」判斷網站每一塊在幫忙還是扯後腿。
2. 用「二、首屏文案公式」寫 hero，用「三、整頁結構藍圖」排版面。
3. 上線前跑一次「四、上線前檢查清單」。

---

## 一、LIFT 逐項檢查

拿網站逐條對照：

| 因子 | 要問自己的問題 | 常見致命錯誤 |
|---|---|---|
| 價值主張 | 首屏 5 秒內講清「是什麼／給誰／解決什麼痛」了嗎？ | 用版本更新或功能清單當主打；用需要先解釋的內部術語當標題。 |
| 相關性 | 訊息跟訪客的來源與期待對得上嗎？ | 對開發者社群用浮誇行銷腔；對一般人堆技術黑話。 |
| 清晰 | 抽象功能有沒有用截圖／GIF／demo「讓人看到」？術語翻成白話了嗎？ | 純文字描述抽象概念；一句話塞三個專有名詞。 |
| 焦慮 | 訪客會有的疑慮（會不會外洩／能不能信／裝起來麻不麻煩）有沒有主動卸下？ | 信任型產品卻沒有信任訊號；網站本身看起來不可信。 |
| 分心 | 首屏「可點的東西」相對「你要他做的那件事」比例接近 1:1 嗎？ | 首屏塞一堆 CTA、導覽、彈窗，稀釋主動線。 |
| 急迫性 | 有沒有一個「現在就行動」的真實理由？ | 用假倒數／假庫存製造急迫（信任型產品尤其致命）。 |

## 二、首屏文案公式（Hero）

首屏 = 標題 ＋ 副標 ＋ 主 CTA ＋ 視覺，四件缺一不可。

**標題（H1）**：一句話講「痛點或成果」，不是講「產品類別」。
- 公式：幫〔誰〕〔達成什麼成果／擺脫什麼痛〕，靠〔關鍵差異〕。
- ✅ 好：「把散落各處的 AI prompt 收進一個本地保險庫，機密只在你同意時才外送。」
- ❌ 壞：「PromptBox v3.2.3 — 機密 Context 保險庫」（版本＋術語，訪客看不懂）。

**副標**：補一句「怎麼做到／為什麼可信」，可嵌關鍵信任詞（本地端、加密、開源）。

**主 CTA**：動詞開頭、只留一個（Download for Mac／View on GitHub）。次要 CTA 最多一個（See how it works）。

**視覺**：放真實產品截圖或 10–15 秒 GIF，別用空泛插畫。抽象功能一定要「被看到」。

> 5 秒測試：找沒看過的人瞄一眼首屏，問他「這是給誰、能幹嘛」。答不出來 = 首屏失敗。

## 三、整頁結構藍圖（軟體／工具類官網通用順序）

1. **Hero**：標題＋副標＋主 CTA＋產品視覺。
2. **信任列（社會證明）**：GitHub star、下載數、社群／媒體提及、「local-first／加密／開源」徽章。
3. **痛點 → 解法**：先講一句訪客會點頭的痛，再講你怎麼解。
4. **核心功能（配圖）**：每個功能 = 一句好處（不是功能名）＋ 一張截圖／GIF；3–5 個就好。
5. **信任／安全專區**（信任型產品必備）：資料流向說明、加密怎麼做、隱私政策、原始碼／安全報告連結、作者是誰。
6. **下載／開始**：清楚標示平台（Mac／Win／Linux）、系統需求、安裝注意（如桌面 App 的「未識別開發者」提示怎麼處理）。
7. **FAQ**：正面回答焦慮（資料會離開電腦嗎？免費嗎？開源嗎？怎麼移除？）。
8. **Footer**：GitHub、文件、授權、聯絡。

> 順序可依產品調整，但 Hero → 信任 → 痛點/解法 → 功能 → 下載 這條主動線盡量保留。

## 四、上線前檢查清單

**價值與清晰**
- [ ] 首屏 5 秒測試通過（陌生人看得懂是什麼、給誰）
- [ ] 標題講痛點／成果，不是產品類別或版本號
- [ ] 每個抽象功能都有截圖／GIF／demo
- [ ] 沒有未經翻譯的內部術語當主打

**信任（信任型產品加倍檢查）**
- [ ] 主要焦慮（資料外洩／可信度／安裝門檻）都有正面回應
- [ ] 網站本身可信：HTTPS、可查到作者、下載來源明確
- [ ] 有原始碼／安全報告／隱私政策連結（若適用）
- [ ] 「資料留在哪、什麼時候會外送」講清楚

**轉換與動線**
- [ ] 主 CTA 只有一個，動詞開頭、位置顯眼
- [ ] 首屏沒有一堆互相競爭的 CTA／彈窗
- [ ] 下載／開始的路徑清楚無歧義
- [ ] 沒有假倒數／假急迫（信任型產品尤忌）

**基本盤（table stakes）**
- [ ] 手機版可正常瀏覽
- [ ] 首頁載入夠快（大圖有壓縮／延遲載入）
- [ ] 連結／下載都能用，沒有壞圖
- [ ] 基本 SEO：title、description、OG 分享圖（分享出去長得對）

## 附錄：範例套用（PromptBox）

- **產品**：AI 工作者用的本地端 prompt 管理／機密 context 保險庫（Electron、靜態加密、當次同意才餵給 AI、卡片串聯、MCP）。
- **產品屬性**：信任型 ＋ 開發者工具 ＋ 免費/開源 → 焦慮與清晰最重、急迫性放掉。
- **Hero 標題示範**：「你的 AI prompt 又多又亂，還常不小心把機密貼進聊天視窗？PromptBox 幫你在本地端收好，機密只在你同意時才外送。」
- **Hero 副標示範**：「本地端、靜態加密、開源可稽核。非經你當次授權，機密不離開這台電腦。」
- **主 CTA**：Download for Mac／View on GitHub（擇一為主）。
- **必放信任區**：安全報告（SECURITY_REPORT）、開源連結、「資料流向」圖、加密說明。
- **功能配圖優先序**：機密卡的鎖定狀態 → consent 釋放對話框 → `{{@prompt:N}}` 展開預覽 → MCP 呼叫透明度。

> 換下一個網站時：只要重填「產品屬性」與「Hero 公式」，其餘結構與清單原封不動即可套用。
