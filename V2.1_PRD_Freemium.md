## 2. 本版核心問題（只寫一個）

如何在不破壞現有使用者體驗的情況下，建立一套可持續運營的收費機制，讓核心功能永遠免費、進階功能提供付費解鎖？

---

## 3. 背景與動機

- **為什麼現在要做這個？**
  V2.0 已完成自動更新管道，代表我們現在有能力安全地向所有使用者推送包含付費牆（Paywall）的新版本，Freemium 的技術條件成熟。
- **使用者遇到什麼痛點？**
  使用者目前所有功能都是免費的，缺乏升級誘因。同時，開發者沒有穩定的收入來源支撐持續開發。
- **不做會發生什麼？**
  軟體長期免費運作有財務壓力，無法維持動力持續迭代高品質功能，最終可能停止維護。

---

## 4. 商業模式設計

### 4.1 策略：Free + Unlock（買斷制）

> **推薦選擇此方案。**
> 訂閱制適合 SaaS 服務端產品（資料在雲端），買斷制更適合本地端應用（資料在使用者自己的電腦）。

| 方案        | 價格                     | 說明                   |
| ----------- | ------------------------ | ---------------------- |
| Free        | $0                       | 永久免費，保留核心功能 |
| Pro（買斷） | 建議 $9.99 USD（一次性） | 永久解鎖所有進階功能   |

### 4.2 功能分層

| 功能                            | Free            | Pro          |
| ------------------------------- | --------------- | ------------ |
| Prompt 管理（新增、編輯、刪除） | ✅              | ✅           |
| 資料夾分類                      | ✅（最多 5 個） | ✅ 無限制    |
| 標籤管理                        | ✅              | ✅           |
| Markdown 預覽                   | ✅              | ✅           |
| 一鍵複製                        | ✅              | ✅           |
| 深色/淺色主題                   | ✅              | ✅           |
| 變數樣板（`{{variable}}`）      | ❌              | ✅           |
| Prompt 數量上限                 | 最多 30 個      | ✅ 無限制    |
| 匯出 / 匯入（JSON/CSV）         | ❌              | ✅           |
| 雲端備份（未來功能）            | ❌              | ✅（規劃中） |

> **⚠️ 設計原則**：Free 版的功能限制要讓使用者「有感但不痛苦」，讓他們可以認識產品並將最核心的工作流建立在上面，但在擴充需求時自然地想到升級。

## 5. 本版目標（Goals）

- **目標 1**：實作本機授權驗證（License Key）機制，能夠識別使用者是否為 Pro 用戶。
- **目標 2**：整合 Lemon Squeezy 付費平台，讓使用者可以透過付款頁面購買 License Key。
- **目標 3**：以優雅、非侵入式的方式呈現付費牆（付費升級提示，不阻擋使用者的正常工作流程）。

**可量化指標**：

- 付費轉換率目標：活躍用戶的 2~5%（業界 Freemium 基準）
- 付費流程完成率：從「點擊升級」到「成功啟動 License」的完成率 > 85%

## 6. 技術架構設計

### 6.1 License Key 運作原理

```
使用者購買 → Lemon Squeezy 發送 License Key（email）
     ↓
使用者在 App 內輸入 License Key
     ↓
App 呼叫 Lemon Squeezy API 驗證 Key 的真實性與狀態
     ↓
驗證成功 → 將 { licenseKey, activatedAt, email } 存入本機加密儲存
     ↓
App 每次啟動時從本機讀取，對照本機狀態決定 Pro 功能開關
```

### 6.2 授權狀態管理（Pinia Store）

新增 `useLicenseStore.js`，管理以下狀態：

```js
{
  isPro: false,          // 是否為 Pro 用戶
  licenseKey: null,      // 已驗證的 License Key
  email: null,           // 購買時使用的 email
  activatedAt: null,     // 啟動日期
}
```

### 6.3 本機儲存方式

- 使用 Electron 的 `safeStorage` API 加密後存入 `electron-store` 或直接寫入設定檔。
- **不存放在 SQLite**（避免與 Prompt 資料耦合）。

### 6.4 付費平台：Lemon Squeezy

**選擇原因**：

- 對個人開發者友善、開箱即用的 License Key 管理
- 支援台灣金流（Stripe 不支援開台灣帳戶）
- 自動處理稅務（VAT/GST）
- 提供 API 驗證 License Key 的有效性

**需要設定的項目**：

1. 在 Lemon Squeezy 建立產品（PromptBox Pro）
2. 設定 License Key 類型（每個 Key 允許啟動 1-2 台設備）
3. 取得 API Key，在 Electron 後端呼叫驗證 API

## 7. 功能需求（逐條寫）

### 功能 1：Pro 功能付費牆顯示

- **目的**：讓使用者清楚知道哪些功能是 Pro 限定，並提供升級入口。
- **使用情境**：Free 使用者觸碰到受限功能時（例如新增第 6 個資料夾）。
- **需求細項**：
  - 在受限功能旁顯示 👑 Pro 圖示（不阻擋，只是視覺提示）。
  - 點擊受限功能時，彈出輕量的「升級提示 Modal」，說明該功能屬於 Pro，提供「了解更多」（連結官網）與「立即升級」按鈕。
  - 「立即升級」按鈕：在系統瀏覽器開啟 Lemon Squeezy 的付款頁面。

### 功能 2：License Key 輸入與驗證

- **目的**：讓購買後的使用者能在 App 內啟動 Pro 授權。
- **使用情境**：使用者購買完成，收到 Email 內的 License Key 後。
- **需求細項**：
  - 在設定頁新增「輸入序號」欄位（只在 Free 狀態下顯示）。
  - 使用者貼上 Key 後，App 呼叫 Lemon Squeezy API 驗證。
  - 驗證成功：顯示「✅ 已成功啟動 Pro 版！」，`isPro` 狀態切換為 `true`，所有 Pro 功能立即解鎖。
  - 驗證失敗：顯示對應錯誤訊息（無效 Key / 已超過啟動上限 / 網路問題）。

### 功能 3：設定頁授權狀態顯示

- **目的**：讓 Pro 使用者清楚知道目前的授權狀態。
- **使用情境**：使用者開啟設定頁面。
- **需求細項**：
  - **Free 使用者**：顯示「免費版 — 升級至 Pro」按鈕。
  - **Pro 使用者**：顯示「👑 PromptBox Pro 已啟動」、購買 Email、啟動日期。
  - 提供「管理授權」連結（連至 Lemon Squeezy 的客戶入口）。

## 8. 非功能需求

- **安全性**：License Key 不得以明文儲存；呼叫 Lemon Squeezy API 的請求須在 Electron Main Process 執行，不直接暴露於 Renderer。
- **離線體驗**：驗證為本機狀態優先，已啟動的授權在離線時仍能正常使用（不需要每次啟動都線上驗證）。
- **防止濫用（初期）**：初期採用軟性驗證（本機 + 偶爾線上驗真），不做硬性 DRM，避免過度工程化。

---

## 9. UI 設計原則

- **克制、非侵入**：Free 使用者打開 App 時「感覺不到付費牆」，只有在他們真正需要 Pro 功能時才看到升級提示。
- **清晰的價值主張**：每個付費牆提示都要清楚說明「升級後你可以得到什麼」，而不只是說「這是 Pro 功能」。
- **快速恢復**：使用者一旦升級，立刻感受到功能解鎖，不需要重啟 App。
