# ADR-001：機密卡片的變數預設值以明文儲存（方案 a）

- **狀態**：已採用（Accepted）
- **決策日期**：2026-07-10（v3.2.2 產品拍板）
- **對應版本**：v3.2.2 機密卡片變數支援
- **相關文件**：[v3.2.2 PRD](../PRD/v3.2.2_PRD_SecretCardVariables.md) · [v3.2.2 Tech Design](../Tech%20Design/v3.2.2_TD_SecretCardVariables.md) §4/§8.5 · [WL-017](../Wishlist/017.v3.2.2_WL_機密卡片變數支援_WL.md)

## 緣起：一個必然會被問的問題

> **Q：為什麼變數面板上寫「變數預設值不加密，請勿在此填入敏感值」？**
> **通常不就是要在這邊放密碼，然後把卡片鎖起來嗎？**

這個直覺非常自然——UI 上掛著 🔒「機密卡片（內容已加密）」，變數面板又是**解鎖後才出現**的，
使用者很容易以為「面板裡的東西也在保險箱內」。但事實恰好相反，而且這種用法是本架構下
**最危險的反模式**。這條警語就是為了打破這個錯覺而存在的。

## 架構事實：這張卡片到底加密了什麼

機密卡片（`is_secret = 1`）在資料庫裡是兩個獨立欄位：

| 欄位 | 儲存形式 | 保護機制 |
|---|---|---|
| `content`（卡片內文） | **密文** | AES-256-GCM，金鑰由使用者密碼經 scrypt 導出；無密碼不可解 |
| `variables`（變數預設值） | **明文 JSON** | 無。與卡片密碼完全無關 |

（全庫層級另有 safeStorage/SQLCipher 的靜態加密，但那是對「整個 DB 檔案」的保護，
與卡片密碼正交——任何能開啟本 App 的人、任何依釋放策略取得輸出的 MCP 客戶端，
看到的 `variables` 都是明文。）

## 正確的心智模型

**敏感值放內文（會被加密）；變數放不敏感的「可調參數」。**

✅ 正確用法——金鑰被鎖住，`staging` 洩漏也無所謂：

```
content（加密）：  "用 API key sk-live-XXXX 查詢 {{env}} 環境的資料庫"
variables（明文）：{ "env": "staging" }
```

❌ 反模式——加密保護到的是那句沒營養的模板，真正的密碼以明文躺在 promptbox.db 裡，
「鎖頭掛在空箱子上」：

```
content（加密）：  "密碼是 {{pw}}"
variables（明文）：{ "pw": "hunter2" }
```

如果你的需求是「存一個密碼」：直接把它寫進內文加密，**不要參數化它**。

## 決策：為什麼不乾脆把預設值也加密？

評估過，有兩道實際的牆（TD §4、§8.5）：

1. **存檔時已經沒有密碼了。**
   解鎖流程是「輸入密碼 → 取得明文 → 密碼立即丟棄」，之後只有明文留在記憶體
   （`useSecretCard` 的安全不變式）。使用者稍後點「儲存」寫預設值時，手上沒有金鑰可用——
   除非每次存檔都重新要求輸入密碼（體驗極差），或把密碼留在記憶體（安全姿態更糟）。

2. **MCP 釋放路徑讀不到。**
   `once` / `session` 釋放策略下，後端是直接從 DB 讀 `variables` 作為 `secretDefaults`
   代入 `compilePrompt` 的（`mcpServer.cjs` 釋放路徑）。若它是密文，`session` 模式的
   後續請求根本無密碼可解，功能直接斷裂。

### 被放棄的替代方案

| 方案 | 內容 | 放棄理由 |
|---|---|---|
| 方案 b：預設值僅存記憶體 | 解鎖後可填值，但不落地 | MCP 釋放拿不到預設值、每次解鎖都要重填，功能價值減半 |
| 方案 c：預設值一併加密 | 進 cryptoService 加密封包 | 撞上前述兩道牆；屬封包語意變更，需連 MCP 讀取路徑一起重新設計 |

### 最終決策（方案 a）

- 變數預設值**明文存 DB**，機密卡解鎖後可持續編輯更新。
- 變數面板**常駐警示**：「變數預設值不加密，請勿在此填入敏感值」（i18n 三語：
  `modal.secret.varPlainWarn`）。
- 設定密碼時，含變數的卡片額外提示「變數面板將於解鎖後才可使用」（`varAfterUnlockHint`）。
- **文件與 UI 的硬規則（PRD §7）：不得在任何介面暗示預設值受加密保護。** 誠實標示邊界，
  而不是給使用者虛假的安全感。

## 後果與界線

- 機密卡片的 `variables` 欄位語意自 v3.2.2 起為「可持續更新的變數預設值」（明文、已警示），
  不再是加密前的凍結快照。
- 卡片密碼保護的**只有內文**。威脅模型上，變數預設值等同一般卡片資料。
- 官網 / 文件描述機密卡片時，必須沿用本 ADR 的心智模型敘述（敏感值進內文、變數是參數），
  不得出現「解鎖後的所有欄位皆受保護」之類的措辭。

## 未來重訪條件

若「加密的變數預設值」需求成真（例如確實要參數化秘密值），屬 TD §8.5 的另立 Wishlist 範疇，
需一次設計三件事，缺一不可：

1. `cryptoService` 加密封包格式變更（content + variables 的聯合封包或雙封包）；
2. 存檔時的密碼取得策略（重新輸入 vs. 會話金鑰的風險取捨）；
3. MCP `secretDefaults` 讀取路徑（尤其 `session` 策略無密碼可用的時點）。

在那之前，警語不是暫時的補丁，而是這個邊界的誠實標示。
