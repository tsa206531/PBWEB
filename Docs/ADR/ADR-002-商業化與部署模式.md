# ADR-002: 商業化與部署模式（Local-first 核心 + 模組化版本）

## 狀態

提議中 (Proposed) - 2026-06-01

> 本 ADR 為策略性決策紀錄，記錄「隱私定位 vs. 成長」三次討論的結論與推論依據。
> 尚未進入實作；目的是把方向固化為可回溯的文件，作為後續 PRD / Tech Design 的上位依據。

## 背景

PromptBox 目前是**單機、單人、本地優先（local-first）**的 Prompt 管理工具：資料存於本機 SQLite，
無登入、無雲端、無多人協作；對外能力集中在本地 MCP 伺服器（綁 `127.0.0.1`）。

在規劃未來方向時出現一個核心張力：

- **規劃中的雲端同步、團隊版**（見 `Doc/web/Architecture_Discussion_2026-03-10.md`、`Doc/web/V2.1_PRD_Freemium.md`）
  能擴大市場與帶來經常性收入，但**與「資料只在你自己電腦」的隱私定位直接衝突**。
- 若放棄成長、純做本地，市場天花板與護城河受限（無網路效應、無經常性收入）。

討論中另浮現兩個關鍵洞見：

1. **本地優先確實是隱私/機密性最強的架構**，但「只剩實體入侵風險」的說法**過於絕對**——
   惡意程式、明文 SQLite、失竊筆電、供應鏈更新皆為非實體的本地風險，其中部分**是軟體該負責的**。
2. **當 LLM 本地化後，PromptBox 可作為地端 / air-gapped 內部 AI 工具的本地上下文供應端**，
   整條 AI 工作流不碰外網——這與隱私定位**一致而非衝突**，且現有 MCP 架構剛好就位。
   但「內部工具」意味著**多人共用**，反而需要 RBAC / 稽核 / 加密（即 PPT03 附錄 B 的企業硬化）。

## 決策

### 決策總綱

**採用「單一 local-first 核心 + 模組化版本（editions / add-ons）」模型，但版本逐一釋出、對外收斂為少數套餐。**

拒絕「純本地（放棄成長）」與「全面轉雲端 SaaS（犧牲隱私）」兩種極端；採用兩者可共存的模組化路線。

---

### 1. 核心原則：可稽核、fail-safe 到本地的核心

- **決策**：保留現有 Electron + SQLite + MCP 作為**單機即完整可用**的核心；任何模組缺失或授權驗證失敗，
  皆安全退回純本地模式。
- **理由**：核心即「隱私版」。最強的隱私宣稱是「聯網程式碼根本不在裡面」，因此隱私版應為
  **編譯時排除聯網模組的乾淨 build**（或 open-core），讓隱私宣稱**可被驗證**，而非 runtime 關閉旗標。

---

### 2. 區分「模式（mode）」與「功能（feature）」

- **決策**：
  - **部署模式（互斥）**：本地（Local）／雲端（Cloud）／地端（On-prem / Air-gapped）。一個 instance 只屬於一種。
  - **跨模式功能（可加購）**：AI（本地 LLM 或自帶 Key）、團隊（RBAC / 稽核 / 共享庫）、進階變數等。
- **理由**：雲端同步與純本地隱私不是「同一個執行實例上可並存的積木」，而是相反的部署姿態；
  把它們當成 mode 才不會產生 `if (isCloud && isLocalPrivacy)` 之類的矛盾狀態。

---

### 3. 對外包裝：收斂為少數套餐，而非無限積木

- **決策**：對使用者呈現 **3–4 個套餐**，而非任意組合的 à-la-carte：
  | 套餐 | 模式 | 主要內容 |
  | :-- | :-- | :-- |
  | Free | 本地 | 核心 Prompt 管理（數量/資料夾上限） |
  | Pro（買斷） | 本地 | 全功能本地 + AI 包 + 變數進階 |
  | Team | 雲端 或 地端 | 共享庫 + RBAC + 稽核 |
  | Enterprise | 地端 / Air-gapped | SSO + 稽核 + 合規 + 地端部署 |
- **理由**：「積木」是**內部架構**的設計手法；給客戶看的**菜單必須小**，否則造成選擇癱瘓與計價/支援災難。

---

### 4. 模組化架構接縫：沿用已規劃的 `services/` 層 + 權益層

- **決策**：模組沿 `Doc/web/Architecture_Discussion` 已規劃的 `services/`（authService / syncService / aiService）邊界生長；
  新增**權益層（entitlement）**判定本安裝可啟用哪些模組。
- **理由**：可插拔的前提是乾淨的模組邊界與權益判定；否則「樂高」會退化成比單體更糟的義大利麵。

---

### 5. 企業路線 = 地端 / Air-gapped，而非雲端 SaaS 優先

- **決策**：企業擴張**優先走地端 / air-gapped + 本地 LLM**（資料留在客戶機房），雲端 SaaS 為可選的 Team 模式之一。
- **理由**：此路線**與隱私定位一致**，把「隱私」從成長天花板轉為進入高合規市場的門票；
  且對隱私敏感組織更好賣（你連碰都沒碰資料）。但須認知：內部多人共用仍需 RBAC / 稽核 / 加密（PPT03 附錄 B）。

---

### 6. 執行順序：先鑿接縫，再逐一出版本

- **決策**：
  1. **先建地基**：自動化測試、權益層、乾淨核心 build。
  2. **先讓基礎版收到第一筆錢**（買斷 Pro 變現上線）。
  3. **由真實付費客戶驗證、一次出一個 edition**，不平行開發三版。
- **理由**：「同時做三個版本」是小團隊的陷阱。模組化的價值在於「接縫先鑿好、之後逐步填」，而非一次全做。

---

### 7. 無論選哪條路都要補的安全洞（純本地框架下亦成立）

- **決策**：優先補兩項，因為它們在「純個人本地隱私」框架下也該做：
  1. **靜態加密（Encryption at rest）**：SQLite → SQLCipher，金鑰以 `electron.safeStorage` 管理。
  2. **MCP token scoping**：單一共用 token → 每人/每角色 scoped token + 速率限制；write-back 需 consent + 權限檢查。
- **理由**：明文 DB 在「筆電失竊」情境下會洩漏全部 Prompt——這**是軟體的責任**，且是隱私版可驗證的賣點。

## 後果

### 正面影響

- ✅ 化解「隱私 vs 成長」假二選一：隱私用戶與成長客群可並存於同一 codebase。
- ✅ 企業路線（地端/air-gap）與隱私定位一致，隱私成為護城河而非阻力。
- ✅ 與既有架構規劃（`services/` 層、`isPro` gating）一致，非另起爐灶。

### 負面影響

- ⚠️ **QA 組合爆炸**：多版本 × 目前零自動化測試 = 回歸風險暴增；測試成為前置而非後補。
- ⚠️ **前期平台工程成本**：權益層、模組邊界、乾淨 build pipeline 都要先付。
- ⚠️ **支援與文件按版本倍增**，對小團隊是隱性殺手。

### 風險

- **隱私品牌可驗證性**：若隱私版與聯網版同一 binary，最在意的客群會質疑「聯網碼真的沒跑嗎」→ 需乾淨 build / open-core 緩解。
- **授權與盜版面**：本地應用授權為軟性,版本價值越高,破解動機越強。
- **市場時機**：企業級本地 LLM 落地仍偏早（估 2–4 年順風），適合「設計時朝此靠」，不適合現在 all-in。
- **執行紀律**：若未守「逐一出版本」原則,易因平行開發而資源稀釋、全部做不完。

---

## 附註

### 被拒絕的替代方案

- ❌ **純本地、放棄成長**：定位最純，但市場天花板與護城河過低。
- ❌ **全面轉雲端 SaaS / 強制帳號**：成長最大，但直接摧毀隱私品牌與既有用戶信任。
