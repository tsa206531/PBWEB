# ADR-003: 產品靈魂與定位（人類策展、AI 取用的機密 Context 保險庫）

## 狀態

提議中 (Proposed) - 2026-06-01

> 本 ADR 記錄一次創辦人 × 顧問深談後對「PromptBox 到底是什麼」的重新定位結論。
> 它是 [ADR-002](ADR-002-商業化與部署模式.md)（商業化與部署模式）的上位文件——先定義靈魂，才談怎麼賣、怎麼部署。
> 屬策略決策，尚未進入實作。

## 背景

PromptBox 至今對外被描述為「本地 AI Prompt 管理工具」。但深談後釐清的真實意圖，與此描述有明顯落差：

### 創辦人意圖（已釐清的【事實】）
- **目標**：融資、做公司、拚規模；認知到此產品**不可能成為獨角獸**（本質是基礎建設），故**最佳歸宿是被收購**。
- **最終使用場景**：在卡片裡放「**只給人設定、機密、密鑰（secrets / keys）**」這類資料——人類策展與管控，AI 連進來「拿完即走」。
- **市場策略**：**鋪向大眾**，而非只打高端。
- **真實受眾與競品（來自實際使用者數據）**：使用者是 **SEO 寫手、AI 影像創作者**；他們的比較對象是
  **Notion / NotebookLM / 記事本**，**不是 PromptHub**。價值點＝「比開網頁找資料、比記事本更方便地專門存 prompt」。
- **成功定義**：有收入養活團隊即算成功（目前僅創辦人一人）。
- **地理**：全球（故已做多語系）。

### 由此產生的定位落差（【推論】）
- 先前的競品分析（見 `PPT/競品分析.md`）以 **PromptHub 為主要對手**——但使用者腦中根本沒有 PromptHub。**瞄錯了靶。**
- 真正的賽道不是「工程師的 Prompt/Skill 工作台」，而是「**非技術大眾的、簡單的、AI 可取用的個人機密 context 儲存**」。

## 決策

### 決策總綱：重新定義產品靈魂

> **PromptBox 不是「更好的 Prompt 管理器」。它是「一個中立、簡單、把人類機密鎖好、再餵給任何 AI 的消費級 context 保險庫」。**
> 人只負責**策展卡片內容**；消費介面刻意留空（介面無關）；產品的靈魂是**把卡片資料鎖好、不外洩**。

---

### 1. 靈魂 = 安全，而非功能；安全是產品本體，不是 infra 衛生

- **決策**：把「機密資料鎖好不外洩」當作**產品核心價值**來投資，而非事後補強的 infra 及格線。
- **理由**：一旦使用場景含密鑰，「鎖好」就是賣點本身。這也是把產品從「比較好用的記事本」升級為
  「**AI 時代的消費級機密保險庫**」的關鍵——後者的故事銳利得多，且「消費級 / AI-native」這個切角
  尚未被 1Password、HashiCorp Vault 等企業級工具佔據，是真空。

---

### 2. 必須先解決「靈魂 vs 現況」的矛盾（最高優先）

- **【事實】現況與靈魂相反**：目前 SQLite 為**明文**（`electron/db.cjs`，非 SQLCipher）、MCP 為**單一共用 token**
  （`electron/tokenStore.cjs`）、走本地 HTTP。**這是「沒鎖」。**
- **決策**：在主打「機密」場景前，下列為**產品本體級**需求（非 backlog）：
  1. **靜態加密**：SQLite → SQLCipher，金鑰以 `electron.safeStorage` 管理。
  2. **MCP token scoping**：共用 token → 每人 / 每角色 scoped，加速率限制。
  3. **密鑰不自動吐出**：機密類卡片**預設不經 MCP 自動回傳**，需**人類明確釋放 / consent** 才釋出。
- **理由**：靈魂與實作今天是打架的；不補等於賣點是假的。

---

### 3. 正視「把密鑰餵給 AI」的攻擊面

- **【推論】**：讓 AI 能 pull 密鑰，本身就是攻擊面——**被注入的惡意 prompt 可使 agent 將密鑰外洩**
  （prompt injection → secret exfiltration）。
- **決策**：機密類資料需有獨立威脅模型——分級（一般 prompt vs 機密）、預設不自動服務機密、釋放需明確同意、
  並記錄釋放軌跡（audit）。**「能被 AI 取用的密鑰」必須當成高風險物件設計。**

---

### 4. 定位選擇：簡單（大眾）> 功能完整（工程師）

- **決策**：以**認知負荷最低**服務非技術大眾為核心定位；不與工程師向工具（PromptHub）比功能堆疊。
- **理由（創辦人論點，採納）**：「Notion 已經這麼簡單，仍一堆人學不會 → 不該高估使用者學習力」；
  工程師是小眾（有需要會自己 DIY，東西開源下載改一改即可）。**簡單本身就是護城河**，
  而工程師複雜介面在大眾面前是劣勢。

---

### 5. 中立的跨平台層 = 最可辯護的護城河（但也最難變現）

- **決策**：押注「**中立、跨平台的 context 互通層**」——大廠只會做**綁自己 IDE** 的基建，
  **結構上不會做**「大家都能一起用」的中立基建，因為那違背其鎖定利益。
- **【價值判斷】**：這是整套策略最強、平台結構上做不到的一塊。但**中立=無鎖定=無定價權**——
  讓你可辯護的東西，同時讓你難收費（見後果/張力）。

---

### 6. 發行：免費版開源 + n8n / MCP 生態整合

- **決策**：**免費版開源**（利於大眾發行與信任）；主動把 **n8n 串好 MCP** 做給社群（公司自己也要用，雙用）。
- **理由**：infra 產品的「發現成本」高；開源 + 自動化生態整合（n8n 社群龐大、受眾重疊）是低成本的漏斗頂端，
  且 n8n 圈同時是發行管道與潛在買家圈。

---

## 後果

### 正面影響
- ✅ 定位從「又一個 prompt 管理器」升級為「消費級 AI 機密保險庫」，故事更銳利、賽道更空。
- ✅ 真實對標 Notion / 記事本（大眾），而非 PromptHub（工程師小眾）——對齊實際使用者。
- ✅ 中立跨平台 + 簡單 + 安全，三者疊加比任一單獨更難被複製。

### 負面影響 / 待解張力（必須選邊）
- ⚠️ **中立 vs 變現**：無鎖定 = 弱定價權。必須明確指出**不破壞中立/隱私的收入線**（密鑰保險庫進階版？團隊？而雲端與隱私衝突，見 ADR-002）。
- ⚠️ **大眾 vs MCP 設定門檻**：最容易的使用者（SEO 寫手）最不可能自己設定 MCP。二選一：MCP 是進階功能（大眾價值僅「比記事本好用」），或把「連上 AI」做到非技術者一鍵。此抉擇決定 roadmap。
- ⚠️ **靈魂 vs 現況**：今天明文 + 共用 token，與「鎖好」矛盾；列為 V3 第一順位而非 backlog。

### 風險
- **安全即賣點 = 安全即責任**：一旦主打機密，任何外洩事件對品牌是致命的（比一般工具嚴重得多）。
- **「簡單」易被複製**：若簡單是唯一護城河，funded 對手可抄；須與中立 + 安全綁定才立得住。
- **被收購的買家未知**：須反推「誰會買、為什麼買」（生產力大廠想加 AI context？平台 acqui-hire？n8n 生態？），再回推現在該累積的資產（使用者數 / 保險庫技術 / 中立品牌）。
