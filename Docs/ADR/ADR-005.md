# ADR-005: 模組化架構接縫與付費版本分層模型（Hexagonal-lite + 刀口 DDD）

## 狀態

提議中 (Proposed) - 2026-06-12

> 本 ADR 收斂 [FP01](../Future%20plans/FP01.md)（架構分層與付費/合規路線討論）與
> [FP02](../Future%20plans/FP02.md)（差距分析與重構規劃）兩份討論，固化為可回溯的決策。
> 屬決策紀錄，尚未實作。上承 [ADR-002](ADR-002-商業化與部署模式.md)（商業化與部署模式）。

## 背景

ADR-002 已定「單一 local-first 核心 + 模組化版本」的**商業策略**，但**未落到程式碼分層**。
FP01/FP02 補上技術面，並回答創辦人兩個問題：

1. 「軟體架構分層怎麼樣？要做付費插件版本該怎麼分？要用 Clean Architecture + DDD 嗎？」
2. 「雲端同步 / 團隊 / 企業 ISO27001・SOC2 / 醫療級這些方向，也是 Hexagonal + DDD 嗎？」

差距分析（FP02 §一）盤出現況是 **Transaction Script**，且有 10 個耦合點阻礙模組化，
其中最關鍵者為 **C2：`mcpServer.cjs` 是與 IPC handler 平行、未經接縫的第二個 DB 存取面**。

## 決策總綱

**採用 Hexagonal-lite（埠與配接器精簡版）+ 只在規則熱點用 DDD 的混合架構；
以「模組契約 + 權益閘門」兩個接縫承載付費版本；以 Strangler Fig 漸進重構既有 core。
拒絕全套 Clean Architecture + DDD（對現規模為過度工程）。**

> 一句話：**別問「Clean Arch 還是 DDD」，問「模組契約長怎樣、權益閘門擺哪裡」。**

---

### 一、架構取向：Hexagonal-lite + 刀口 DDD（拒絕全套 DDD）

- **決策**：只取 Clean Architecture 裡「製造接縫」那塊（依賴反轉 + 薄 use-case 層 + repository 抽象），
  丟掉 DDD 儀式（Aggregate / VO / 領域事件）。DDD **只**點在 4 塊規則熱點：
  變數同步、串聯展開（`schemaUtils.cjs` 已是範本）、機密釋放策略、權益判定。
- **理由**：桌面 core 八成是 CRUD，領域不夠複雜到需要被 DDD 馴服；全套會拖慢第一個付費版本。
  抽純函式的真正動機是**可單元測試**，不是純度。

### 二、兩個接縫：模組契約 + 權益閘門

- **決策**：
  - **模組契約（Module Contract）**：每個功能（含 core）是自我描述單元，可向 registry 註冊
    `{ id, requires, ipc, migrations, mcpTools, register(ctx) }`。
  - **權益閘門（Entitlement Gate）**：registry 註冊前判定 `isEntitled(requires)`；未授權 → 整包不掛，**安全退回純本地**（兌現 ADR-002 §1 fail-safe）。
  - **core 與 add-on 用同一套契約寫**，不養雙軌技術債。
- **理由**：可插拔的前提是乾淨模組邊界 + 權益判定，這比分層名詞重要。

### 三、版本模型：mode / feature / compliance 三軸（破除「四個並列版本」的誤會）

- **決策**：把 FP01 §六的洞見定為正式分類——使用者提的方向**不是同一種生物**：

  | 軸 | 定義 | 例 | 互斥? |
  | :-- | :-- | :-- | :-- |
  | **部署模式 (mode)** | 改變拓樸（client/server 邊界） | 本地 / 雲端同步 / 團隊 / 地端 | 互斥（一 instance 一種） |
  | **跨模式功能 (feature)** | 可加購能力 | AI 包、進階變數 | 可疊加 |
  | **合規規範 (compliance)** | 控制項 + 流程 + 證據，疊在 mode 上 | ISO27001 / SOC2 / HIPAA / IEC62304 | 非「版本」 |

- **理由**：ISO27001/SOC2/醫療**不是你「架構」出來的 codebase**，是「營運通過稽核的服務 / 跑受監管流程」。
  把它們當第 4、第 5 個 edition 開發，是燒掉小團隊的典型誤會。
  合規架構的任務**只是把控制項接縫（稽核日誌、RBAC、加密）留乾淨**，其餘 80% 是流程與證據。

### 四、付費版本打包：A 乾淨 build / B 權益旗標 / C 永不做

- **決策**：
  | 做法 | 用於 | 隱私可驗證性 |
  | :-- | :-- | :-- |
  | **A. 編譯期乾淨 build** | 隱私版 / 企業地端 / 醫療地端 | 🟢「聯網碼根本不在 binary」可驗證 |
  | **B. Runtime 權益旗標** | 跨模式 feature（AI 包等本就需聯網） | 🔴 弱（程式都在） |
  | **C. Runtime 動態載入第三方插件** | **永不採用（早期）** | 授權/安全/QA 三重地獄 |
- **理由**：A 與 B 共用同一模組契約，差別只在「被編譯排除 vs 被權益關掉」。要的是「第一方模組於 build/授權時選擇」，不是開放外掛市集。

### 五、Hexagonal 與 DDD 在四方向的適用度（回答問題 2）

- **決策**：
  - **六角架構：全程適用且回本加倍**——為 add-on 鑿的 repository 埠，到 client/server 拆分時原地升級成網路邊界（本地接 SQLite 配接器、雲端接 HTTP 配接器，領域邏輯不改）。
  - **DDD：桌面 core 維持刀口式；到團隊版 server 側才全面回本**——多租戶/RBAC/稽核有真 invariant 與統一語言，屆時值得真的劃 bounded context（Identity&Access / Prompt / Sync / Audit&Compliance）。
  - **團隊領域寫成 deployment-agnostic**：同一份 server code，配接器一換即可雲端 SaaS **或** 客戶地端/air-gapped——讓「醫療版＝團隊版＋地端＋醫療控制項＋IEC62304 流程」，而非另一條 codebase。
- **理由**：此舉讓隱私定位（資料不離開客戶機房）從成長天花板轉為高合規市場門票（呼應 ADR-002 §5）。

### 六、執行：Strangler Fig 漸進重構，Go/No-Go 分級

- **決策**：依 FP02 §四，分 A 前置解耦 / B 核心重構 / C 權益整合三階段，每步行為等價、可獨立 merge、測試綠燈。並採 FP02 §5.3 的 Go/No-Go：
  1. **強烈建議做（即使不賣插件也淨賺）**：補測試網 + repository 抽取 + **B1（MCP 改用 repository，消滅 C2 雙實作）** + migrator 版本表。
  2. **看商業確定性才做**：registry / entitlement / preload 泛型橋（確定要出付費模組才回本，否則 YAGNI）。
  3. **明確現在不做**：server 側 DDD、雲端、合規控制項——屬 greenfield，待第一個付費模組收到錢、契約被真實驗證後另立 PRD（呼應 ADR-002 §6「不平行開發」）。
- **理由**：B1 是最高風險但效益最高的一步，已獨立成 Tech Design（見附註）先行；其餘按商業確定性逐一填。

---

### 七、未決議題與信任邊界（FP02 critical pass 缺口回填）

> 本節回填 [FP02](../Future%20plans/FP02.md) 評估後仍未在前六節定案的議題。Gap 0（順序矛盾）影響最大，已獨立評估為 [FP03](../Future%20plans/FP03.md)。

- **(信任邊界) 權益 gating 採 defense-in-depth**：renderer（`licenseStore`）只負責「藏 UI / upsell」，**永不可信**；真正的能力閘門在 main process 的 registry——**未授權的模組，其 IPC handler / MCP tool 根本不註冊**，而非註冊後再 `if` 擋。理由：renderer JS 使用者可改，唯一可信閘門在 main。

- **(離線授權) 不追求「離線可驗證 + 不可破解」**：此與隱私品牌（不打電話回家）成數學三角衝突。分流——Pro 買斷走**弱授權，接受一定比例被破解**；企業/醫療版的真正防線是 §四 的**做法 A 乾淨 build**（破解面與動機本就不同），而非授權碼強度。待補：量化「破解容忍度 vs 價位」的威脅模型。

- **(契約落差，標記限制) 模組契約的 `mcpTools` 目前是空頭支票**：`mcpServer.cjs` 寫死 prompt-backed 工具（`prompt_${id}`），尚無「非 prompt 工具」注入接口。模組契約 TD 須明列此限制，並補 `contractVersion` 相容政策與**模組間 migration 拓樸排序**（B 模組的表依賴 A 模組的表時）。

- **(測試矩陣) edition 測試採分層，避免組合爆炸**：**每模組對 `ctx` 的契約測試 + 少數端到端 edition smoke**，不做 Free × Pro × entitled 的全組合測試。

## 後果

### 正面影響

- ✅ 把 ADR-002 的商業策略落成可執行的程式碼接縫，非停在策略層。
- ✅ repository 收斂 IPC 與 MCP 的雙實作（C2），**即使不賣插件也淨賺**（DRY + 可測 + DB 可維護）。
- ✅ 三軸分類避免把合規當「版本」去開發，省下小團隊最易踩的坑。
- ✅ 六角接縫一路複用到 client/server，投資複利。

### 負面影響

- ⚠️ 前期平台工程成本（registry / entitlement / migrator）需先付。
- ⚠️ B1 同時牽動「GUI 取卡」與「AI 經 MCP 取卡」兩路徑，回歸面大。
- ⚠️ 零自動化測試是硬成本前置（補測試網約 2 人天，不可省）。

### 風險

- repository 行為若未以「IPC ∪ MCP 現行行為聯集」為驗收基準 → 兩路徑分歧（見 B1 TD 的退路）。
- migrator baseline 認定錯誤 → 砸壞既有使用者加密 DB。
- 範圍蔓延：手滑去蓋 server/DDD → 違反本 ADR §六.3，PR 應拒收。

## 附註

### 被拒絕的替代方案

- ❌ **全套 Clean Architecture + DDD**：對 100 筆 prompt 的桌面 core 為過度工程，拖慢變現。
- ❌ **維持 Transaction Script 不動**：無法承載可插拔付費模組，且 C2 雙實作技術債持續累積。
- ❌ **Runtime 第三方插件市集（FP01 做法 C）**：授權/安全/QA 三重地獄。
- ❌ **把 ISO27001/SOC2/醫療當並列 edition 開發**：誤把合規 regime 當 codebase。
