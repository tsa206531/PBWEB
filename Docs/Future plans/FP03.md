# FP03：第一筆營收的順序決策 — 最小可收費路徑 vs 先重構（Gap 0）

## 文件資訊

- **日期**：2026-06-12
- **類型**：Future Plan（順序決策評估，尚未執行）
- **觸發**：[FP02](FP02.md) critical pass 的 **Gap 0**——「先重構」與「先收錢」順序矛盾
- **上位**：[ADR-005 §六/§七](../ADR/ADR-005.md)、[ADR-002 §6](../ADR/ADR-002-商業化與部署模式.md)
- **狀態**：待創辦人拍板

---

## 一、問題：兩份文件自相矛盾

| 來源 | 主張 |
| :-- | :-- |
| **FP02 §四**（隱含） | 先做 repository / B1 / migrator / registry / entitlement（≈18 人天）→ 才掛第一個付費模組 |
| **ADR-002 §6.2**（明文） | 「**先讓基礎版收到第一筆錢**（買斷 Pro 變現上線）」 |

> 矛盾核心：**18 人天的內部管線不產生任何使用者價值、不帶來一毛營收。**
> 在「真的有人付錢」被驗證**之前**就投資地基，是單人/小團隊最容易耗盡現金流與動力的方式。

這不是技術問題，是**順序與下注時機**問題。

---

## 二、兩條路徑

### Path R — 先重構後變現（FP02 的隱含假設）

```
A 補測試 + repository → B1 MCP 收斂 → B2 migrator → B3 registry → C entitlement → 第一個付費模組
└──────────────────────── ≈18+ 人天，零營收 ────────────────────────┘
```

| 面向 | 評估 |
| :-- | :-- |
| 優點 | 第一個付費模組上線時地基乾淨、契約已驗證、後續每個 edition 便宜；不留 gating 技術債 |
| 缺點 | 18+ 人天零營收零使用者價值；registry/entitlement 是**對「會有人付」的投機下注**；現金流與動機風險高 |
| 最大風險 | 做完才發現**沒人願意付** → 投機部分（registry/full entitlement）幾乎白做；中途失去動力 |
| 可逆性 | 低（已投入大量時間） |

### Path M — 最小可收費路徑先行（ADR-002 §6.2 的字面意思）

```
一個 main 端 entitlement 檢查 + 離線授權碼弱驗證 + 付款連結 + 輸入 key 的 UI → 收第一筆錢 → 看轉換率再決定是否投資重構
└──────────────── ≈2~3 人天 ────────────────┘
```

| 面向 | 評估 |
| :-- | :-- |
| 優點 | 最快驗證 willingness-to-pay；現金流與心理正回饋；避免為不存在的需求過度建設 |
| 缺點 | gate 是 throwaway 技術債；在未重構 code 上加 `if (isPro)` 會短暫污染 |
| 最大風險 | 對既有用戶加 paywall 的反彈（緩解：grandfather 既有用戶 / cap 設高 / 只對新功能收費） |
| 可逆性 | **高**（沒人付 → 移除一個 `if`，近乎零沉沒成本） |

---

## 三、第三條路（推薦）：Hybrid — 最小可收費，但 gate 種在「會保留的接縫」上

Path M 的技術債之所以可怕，是因為「hacky gate 之後要整個重做」。但這可以避免：

> **第一筆營收走最小 gate，但 gate 寫在『未來 registry 也會用的同一個 entitlement 接縫』上**——
> 落地一個最小的 `entitlement.isPro()`（main process、ADR-005 §七 信任邊界），在現有 handler 註冊處包一層。
> 這**不是**完整 registry，但是 registry 的**種子**，不是純丟棄品。

同時，把 FP02 §5.3 已分好的兩類重構**與營收解耦**處理：

| 重構類別 | FP02 §5.3 定性 | 與營收的關係 |
| :-- | :-- | :-- |
| **B1 / repository / migrator** | 「即使不賣插件也淨賺」（DRY+可測+DB 可維護） | **與營收解耦**：本來就值得做，平行慢慢做，但**不是營收前置** |
| **registry / full entitlement** | 「看商業確定性才做」（投機） | **營收後置**：等第一筆營收/轉換數據再投資 |
| **第一筆營收的 gate** | 新增 | **走最小接縫**，種在未來 registry 會用的 `entitlement` 上 |

這恰好**調和了矛盾**：Path R 與 Path M 的對立，來自把「淨賺重構」和「投機重構」綁成同一包。拆開後——淨賺類可做但非前置、投機類營收後置、第一筆錢走最小 gate。

## 四、最小可收費路徑：候選與粗估

| 候選 | 內容 | 粗估 | 驗證的問題 |
| :-- | :-- | :-- | :-- |
| **候選 1：Free 上限 + 授權解鎖** | 對 prompt/folder 數設軟上限，授權碼解鎖；近乎零新功能工程 | ≈2~3 人天 | 「**願不願意為這工具本身付錢**」 |
| **候選 2：AI 包**（ADR-002 Pro 內容） | 真接 AI API + key 管理（`aiService` 已規劃） | ≈5~8 人天 | 「**願不願意為 AI 功能付錢**」 |

> 兩者驗證的是**不同問題**，先做哪個取決於你想驗證的價值主張。
> 若目標是「最快拿到願付訊號」→ 候選 1；若「AI 是核心賣點」→ 候選 2。
> 共同前提：付款用現成金流（Gumroad/Stripe Payment Link），**不自建後端**（呼應 ADR-002 不自建）。

## 五、決策準則（什麼情況選哪條）

| 若…… | 則 |
| :-- | :-- |
| 「是否有人付錢」**尚未驗證** | → **Hybrid / Path M**（先驗證，別猜） |
| 已有強付費訊號（waitlist / 預購 / 明確客戶） | → 可直接 Path R 投資地基 |
| 現金流 / 動機是約束 | → **Path M** |
| 團隊≈單人 | → **Path M / Hybrid**（單人扛不起 18 人天無回饋） |
| B1/repository 已因「淨賺」理由想做 | → 照做，但**標記為與營收無關**，別記在「變現前置」帳上 |

---

## 六、建議

**走 Hybrid：**

1. **先做最小可收費**（候選 1 或 2 擇一），gate 種在 `entitlement.isPro()`（main、defense-in-depth）。上線、接金流、**收第一筆錢**。
2. **B1 / repository / migrator 與營收脫鉤**，當「降技術債」獨立進行——做不做都不卡變現。
3. **registry / 完整 entitlement / 多 edition** 等到第一筆營收與轉換率出來，用**真實數據**決定是否投資（避免對不存在的需求過度建設）。
4. server 側（雲端/團隊/合規）維持 greenfield 後置不變。

> 一句話：**別讓漂亮架構擋在第一筆營收前面。** 先用最小接縫收到錢、驗證有人付，再回頭把地基蓋厚——而那條最小接縫，剛好是未來 registry 的種子，不浪費。
