## 〇、一句話

> **私有源碼 + 第一階段付費功能全是純本機 + 要無痛升級** → 授權走 **approach B（同一個 app + 離線簽章碼 + main-process gate）**，**不是 A（分版 binary）**。
> 免費版永遠是基底、Pro 就地解鎖、資料零遷移。雲端／團隊是**另一隻怪獸**——另立、訂閱制、且「雲端 ≠ 團隊」。

---

## 一、本次討論確立的前提

1. **源碼私有（閉源）** → runtime 旗標不會「被人讀 GitHub 改一行」就破（但 Electron asar 仍可解包，見 §六）。
2. **第一階段付費功能全是本機**：更多卡片 + 卡片串聯 + 強型別。**不聯網。**
3. **尚未公開發佈、私人倉庫、無任何使用者**（GitHub release 在私倉；強型別/串聯雖已做好，但沒人在用）→ **這是畫 Free/Pro 界線最便宜的時機**：從第一個公開版就把線定死，**之後永遠不會踩 clawback（收回已免費功能）**。鐵律仍成立（公開後就不能再收回），但因尚未公開，現在可乾淨設界、不傷任何人。

---

## 二、授權機制：為何是 approach B，不是 A

|                               | **A：分版乾淨 build**                                                                        | **B：同一個 app + 翻旗標**（✅ 本案）      |
| ----------------------------- | -------------------------------------------------------------------------------------------- | ------------------------------------------ |
| 付費碼                        | 編進另一個 binary                                                                            | 都在，授權碼解鎖                           |
| 升級資料連續性                | ❌ 不同 appId/userData/safeStorage scope → **既有加密 DB／機密卡可能解不開**（= 密碼鎖災難） | ✅ 同 DB、同加密範圍，就地解鎖             |
| 隱私賣點（聯網碼不在 binary） | 對**聯網**付費功能成立                                                                       | 本案付費功能**全本機**，A 的隱私好處用不上 |
| 適用                          | 隱私/企業地端版、聯網模組                                                                    | **純本機加值功能（本案）**                 |

**結論**：本案付費功能全本機、且要無痛升級 → **B 完勝**。A 會親手製造創辦人擔心的「重構前後密碼鎖」資料連續性問題（不同 app 身分 → 加密金鑰範圍不同 → 機密卡解不開）。

---

## 三、不聯網怎麼發權益（具體流程）

1. **簽章授權碼**（建議 Ed25519）：你的**私鑰**簽、app 內嵌**公鑰**、**main process 本地驗章** → 通過才翻 entitlement 旗標。**全程不聯網。**
2. **收錢＋發碼**：用 Gumroad / Lemon Squeezy / Paddle，購買當下自動發碼／發下載連結，**不需自架伺服器**。
3. **升級 UX**：`Settings → 輸入授權碼` → main 驗章 → 旗標翻開 → 卡數上限解除、串聯/強型別 UI render。**資料零遷移、密碼鎖不受影響。**

> 這就是「不聯網怎麼讓人用到權益版」的答案：**同一個 app、貼一組離線可驗的簽章碼、main 本地驗、就地解鎖。** 不需 server、不需分版 binary。

## 四、免費 / Pro 邊界

| 功能                 | 免費                                 | **Pro（$20，本機全解鎖）** |
| -------------------- | ------------------------------------ | -------------------------- |
| 卡片數（總數）       | 30 張（main 擋第 31 張）             | 解除上限                   |
| 變數 Variables       | ✅                                   | ✅                         |
| MCP                  | ✅                                   | ✅                         |
| **密碼鎖 / 機密卡**  | ✅（安全原語，**不 gate**）          | ✅                         |
| **卡片串聯**         | ✅，但 chain 資料夾**最多 5 張卡片** | 無限                       |
| 強型別 Strong Typing | ❌（**完全隱藏**）                   | ✅                         |

> 兩個付費功能 gate 方式**不同**：強型別走「**布林隱藏**」、串聯走「**配額**」。原因見 4.1。

### 4.1 gate 機制（已拍板）

- **單一 entitlement 真相**：一個 `pro` 旗標。**首發前 = 寫死 `false`**；Pro 上線後 = `licenseStore.pro`（＝ main 驗章結果）。**碼留在 binary，不刪、不分版**（approach B）。
- **強型別 = 布林隱藏（(b) 決策）**：`SchemaPanel` 的強型別 toggle 只在 `pro` 為真時 render → 免費**完全看不到「強型別」這個詞**，`enableStrongTyping` 永遠 false → `schema_definition` 永遠 null。**不做 per-variable 配額**（同卡內「有些變數設了型別、有些被鎖」的半殘狀態太醜，見 WL-016）。renderer 隱藏 + main **拒寫 `schema_definition`**（縱深防禦）。靠**獨立升級頁**讓人發現。順帶解決先前「強型別視覺安靜化」。
- **串聯 = 配額（chain 資料夾 ≤ 5 卡）**：免費**可以用**串聯，但 chain 資料夾內最多 5 張卡片。**main process 在「新增/移入 chain 資料夾」時擋第 6 張**（`folder_id = chain` 計數），Pro 解除。配額單位是「**整張卡**」（乾淨、非半殘），正是可接受的 freemium 嘗鮮形式；撞上限時自然帶出升級提示。
- **卡片數 = 配額（總數 ≤ 30）**：main 擋第 31 張，Pro 解除。
- **gate 兩層**：renderer 不 render UI（門面）＋ **能力擋在 main process**（真鎖）。renderer gating 永不可信（呼應 FP01/FP02）。
- **資料安全鐵則（呼應 §5.5）**：所有配額只擋「**新增 / 移入**」，**永不刪除或隱藏既有卡片**。若某情境出現 chain 資料夾已 >5 或總數 >30，一律 **grandfather**（保留、僅擋再加），不毀資料。

## 五、密碼鎖 / 資料連續性 —— 真正的紅線（grounded in `db.cjs`）

> 更正先前過於武斷的「重構不動 schema」。精確版如下。

### 5.1 兩條軸不要混

| 軸                                          | 會發生什麼                                     | 對密碼鎖                       |
| ------------------------------------------- | ---------------------------------------------- | ------------------------------ |
| **免費 → 付費**（同版本、翻旗標）           | 同 app、同 DB、**不觸發任何 migration**        | **零影響**                     |
| **版本升級**（v3.2 → 未來含 FP02 migrator） | schema **會多一張 `schema_migrations` 版本表** | 既有欄位/加密層不變 → 資料連續 |

「重構前後的密碼鎖」要擔心的是**版本升級**那條，跟**買不買 Pro 無關**。

### 5.2 現況兩層加密（都不碰）

- **DB 層 SQLCipher**：`db.cjs:64` `pragma key = x'…'`，金鑰來自 keyService（safeStorage 存）。
- **卡片級加密**：機密卡密文存 `prompts.content`、metadata 存 `prompts.enc_meta`（`db.cjs:122`），由 cryptoService 處理，**與 DB 層獨立**。

### 5.3 FP02 重構實際動到的 vs 不動的

| 動                                                                          | 不動                                                                       |
| --------------------------------------------------------------------------- | -------------------------------------------------------------------------- |
| 新增 `schema_migrations` 版本表（取代 `db.cjs:117-146` 的 try/catch ALTER） | 既有表欄位格式（prompts/folders/settings/mcp_tokens/secret_release_audit） |
| 現況 schema 認定為 baseline v0                                              | SQLCipher 金鑰處理、safeStorage 金鑰來源                                   |
| migration **機制**翻新                                                      | 卡片級加密、`enc_meta` 格式、密碼鎖密文格式                                |

### 5.4 真正要守的紅線（不是「schema 不變」）

1. **SQLCipher 金鑰處理 + safeStorage 金鑰來源不變**；
2. **既有欄位格式不變**（`content` / `enc_meta` / `schema_definition` / `variables` …）；
3. **migrator 正確認 baseline v0、不重跑 ALTER**（FP02 已標 R2 風險，配套：冪等 + 失敗即不寫版本 + 拿真實舊 DB 測 + 保留 `db.cjs:44` 的 `.plain.bak`）。

### 5.5 資料安全鐵則

> Entitlement 閘門只能擋「**新增 / 使用**」，**永不刪除或隱藏既有資料**。
> 30 卡上限 = 擋「加第 31 張」；對既有 >30 張的使用者 **grandfather**（保留、頂多唯讀不能再加），**不得因權益判定毀資料**。違反它＝懲罰最早的支持者。

## 六、威脅模型（誠實）

- **Electron 的 JS 打包進 `app.asar`，使用者磁碟上就有，可解包讀。** 私有源碼擋「直接讀 GitHub 改」，但擋不掉解包 patch。
- **簽章**能擋偽造金鑰 / keygen（沒你私鑰簽不出有效碼），但**擋不掉**有心人把 `isEntitled` patch 掉。
- 對策：gate 放 main、bundle minify、必要時關鍵檢查塞 native 模組（重，通常不值得）。
- **結論：離線 ⊥ 防破解。接受會有漏，為誠實使用者優化，別做 Denuvo。** 生產力小工具，大多數人不會為省 $20 去 patch asar。

---

## 七、雲端 / 團隊：另一隻怪獸（$35 tier 要先想清楚）

1. **打破「不聯網」前提**：雲端/團隊**必須後端伺服器**（同步、帳號、多租戶）→ 權益改走**線上帳號**（本來就在線上，比離線簽章更易更穩），與本機 Pro 的離線授權**是兩套**。
2. **一次性 $35 賣「持續成本」的服務＝危險經濟學**：雲端每個重度用戶都是你的**長期成本**，一次買斷 → 用越久越虧 → 幾乎一定要**訂閱制**。
3. **「雲端 ≠ 團隊」size 差很多**（FP01 §6）：雲端同步（單人、跨裝置備份/還原）相對可控；**團隊**（多人/RBAC/多租戶/衝突/稽核）是大跳躍、要你營運 SaaS。別綁成一個 tier。
4. **分期建議**：先驗證 **Pro $20 有沒有人買** → 雲端（**訂閱**）→ 團隊（更後，且看你願不願當 SaaS 維運者）。$20→$35 只差 $15 卻跨到「要養伺服器」的等級，數字先別急著定。

## 八、Go-to-market 順序（接 FP03，選最小可收費路徑）

1. **現在**：免費版照常發、繼續把免費核心做好（漏斗 + 信任 + 永久免費基底）。
2. **隨時（低風險）**：FP02 的「淨賺」重構（repository / 單一真相 / 接縫）——**不綁商業承諾**，做了對誰都好。
3. **有第一個付費價值 + 商業決心**：只為 Pro 建**最小**機制 = **驗章 + 3 個 gate（卡數 / 串聯 / 強型別）+ 升級頁**，用 storefront 開賣。**不為此蓋整個 registry/模組契約平台。**
4. **多個付費模組被真實銷售驗證後**，才建完整 entitlement 平台（FP02 B3/C）與「Settings 作為控制中心」。

> 呼應 FP03：走 **Path 最小可收費**，不走 Path R（先重構 ~18 人天零營收）。**永遠不需要「停下來全重構才能開賣」，也不需要把現有版本改成權益版丟給用戶。** 增量加付費模組，第一個就能收錢。

## 九、決議與待決

### 9.1 已決議（2026-06-24）

- **首發前無公開用戶 → Free/Pro 界線從第一個公開版就定死**（私倉、強型別/串聯尚無人用）。畫線最便宜的時機，**之後永遠不踩 clawback**。
- **三條 gate（見 §四 4.1）**：強型別 = 完全隱藏（布林）；串聯 = chain 資料夾 ≤ **5 卡**（配額）；卡片數 ≤ **30**（配額）。
- **第一波免費版不做 Pro 機制、不做試用配額**：把 `pro` 旗標寫死 `false` 即可發布。內容 = **MCP 活動紀錄（WL-009）+ 技術 Log（WL-010）+ UI 收尾 + 上述 gate**。強型別/串聯的碼留在 binary、gate 起來，不刪。
- **試用 / Pro 展示頁屬「Pro 上線後」的轉換槓桿**：若做試用優先「**時間限定全解鎖**」；若做配額，單位用「**整張卡**」（如串聯 5 卡），**不用「N 個變數」**（半殘狀態太醜）。

### 9.2 待決（給創辦人）

1. **Pro 買斷 vs 一次性**：建議**永久買斷**（本機功能無持續成本，買斷對隱私品牌也乾淨）。
2. **選型**：簽章方案（Ed25519 + 內嵌公鑰）、金流（Gumroad / Lemon Squeezy / Paddle）。
3. **升級頁 UX**：獨立區、非 inline 灰色誘餌（呼應 §四）。
4. **雲端 / 團隊另立**：訂閱制、線上權益、雲≠團隊，需獨立 PRD。
