## 〇、範圍界定（先講清楚這份報告「不」做什麼）

FP01 描述的終局橫跨「桌面 core 模組化」到「團隊版 server 側 DDD / bounded context」。
但 server 側（雲端/團隊/合規）目前是 **greenfield（綠地）**，沒有既存程式碼可「重構」。

> **本報告只針對「現有桌面 core，重構到可承載模組契約 + 權益閘門」這一段**——也就是 FP01 §十的步驟 1~3。
> server 側 DDD 不在本次重構範圍（屬新建，待第一個付費模組驗證契約後另立 PRD）。

這個界定很關鍵：**先把接縫鑿在既有 code 上，不要在沒有付費客戶前去蓋 server。**

## 一、現況基準：實際耦合點盤點（grounded）

| # | 現象 | 證據（檔案） | 對 FP01 的阻礙 |
| :-- | :-- | :-- | :-- |
| C1 | IPC handler 直接 `db.prepare()`，傳輸+業務+SQL 焊死 | `handlers/promptHandlers.cjs`、`handlers/folderHandlers.cjs` | 邏輯無法當模組抽走 |
| C2 | **MCP server 是第二個未接縫的 DB 存取面**，平行重複 SQL + 業務規則 | `mcpServer.cjs:150-328`（直接查 folders/prompts、內嵌串聯展開、機密釋放、稽核寫入） | 同一規則兩處實作，改一處漏一處 |
| C3 | 註冊是命令式、寫死 | `ipcHandlers.cjs:6-15`（`setupPromptHandlers / setupFolderHandlers / setupMcpHandlers` 硬呼叫） | 無 registry，模組不能自我註冊 |
| C4 | 無權益判定 | 全 codebase 無 entitlement / license / isPro | 付費模組無閘門可掛 |
| C5 | DB schema 集中、migration 是 try/catch 硬加欄位、無版本表 | `db.cjs:87-171`（一坨 `CREATE TABLE` + `migrations[]` 靜默吞 duplicate column） | 模組無法擁有自己的表/migration |
| C6 | DB 集中擁有「本應屬於模組」的表 | `db.cjs:148-171`（`mcp_tokens`、`secret_release_audit` 與 core prompts 混在同一處） | 模組與 core 的資料邊界糊掉 |
| C7 | 領域邏輯埋在 handler，未抽純函式 | `promptHandlers.cjs:10-39`（`_extractVariables` / `_syncAndStringifyVariables`） | 不能單元測試、不能跨 IPC/MCP 共用 |
| C8 | **變數抽取邏輯重複實作** | `promptHandlers.cjs:_extractVariables` ⟷ `src/utils/variables.js` | DRY 破口，前後端可能漂移 |
| C9 | preload 橋接靜態寫死 | `preload.cjs:5-63`（`electronAPI/mcpAPI/updaterAPI` 三個固定物件） | 模組新增 IPC channel 無法動態進 preload |
| C10 | renderer store 無權益/授權狀態 | `src/stores/`（promptStore/mcpStore/filterStore/preferenceStore，無 licenseStore） | 前端無法依權益顯示/隱藏功能 |

**好消息（已是接縫雛形，重構時保留並當範本）：**
- `schemaUtils.cjs`：`expandChains / deriveSchema / validateArguments / compilePrompt / ChainError` —— 已是無 IO 的純領域邏輯，**這就是 FP01 §五要的「刀口 DDD」的正確長相**。
- `services/`：`cryptoService / keyService / consentBroker` —— 已是乾淨的橫切服務。
- `main.cjs` 啟動序列定義清楚（key→DB→MCP→handlers→tray），重構時只需在中間插入 registry。

## 三、影響範圍與依賴分析（Impact Scope & Dependencies）

### 3.1 系統邊界：會被波及的相鄰功能

```
            ┌─────────────────────────────────────────────┐
   repository 抽換 ──┬──► IPC handlers（prompts/folders）   │ 一改
                     └──► MCP server（resources/tools/call）│ 兩個消費者
            ┌─────────────────────────────────────────────┘
   migrator 引入 ───► db.cjs 啟動序列 ───► 既有使用者 DB（向後相容風險）
   registry 引入 ───► main.cjs 啟動序列 ───► tray / consentBroker 對主視窗參照
   preload 改動 ───► 全部 renderer 呼叫（electronAPI.* / mcpAPI.*）
```

**關鍵牽連點：**
1. **MCP server 與 IPC 共用 repository** —— 這是最大收斂效益（C2 消滅），但也意味著 repository 的任何行為差異會**同時**影響「使用者 GUI」與「AI 經 MCP 取卡」兩條路徑。repository 必須以兩邊現有行為的**聯集**為驗收基準。
2. **機密卡片邏輯橫跨 IPC + MCP + consentBroker** —— `release_policy` 分支同時存在於 `promptHandlers.cjs`（set/clear/unlock）與 `mcpServer.cjs:255-299`（consent 釋放 + 稽核）。抽領域層時必須兩路徑同步，否則出現「GUI 能解、MCP 不能解」或稽核漏記。
3. **串聯資料夾守護**（`folderHandlers.cjs:41-51` + `db.cjs:191-202`）—— 串聯資料夾 id 是「單一真相寫進 settings」的隱性契約，folderRepository 重構不可破壞此守護。

### 3.2 耦合風險與潛在副作用（Side Effects）

| 風險 | 機制 | 潛在副作用 |
| :-- | :-- | :-- |
| **migration 向後相容** | 既有使用者 DB 已用 try/catch 加過欄位；引入版本表時，需把「現況視為 baseline 版本」 | 若 baseline 認定錯誤 → 重跑 ALTER → 噴錯或資料異常 |
| **SQLCipher 開庫順序** | `main.cjs:106-107` key→initDB 不可調換；migrator 必須在開庫**之後** | 順序錯 → 對加密庫跑 DDL 失敗 |
| **MCP stateless 每請求建 server** | `mcpServer.cjs:399-424` 每個 `/mcp` 請求 new 一個 server 實例 | repository 若持有可變狀態/快取 → 跨請求污染；repository 必須無狀態或以 db 為唯一狀態 |
| **preload 泛型 invoke 的安全面** | 開放 `invoke(channel, payload)` 會擴大 renderer→main 的攻擊面 | 必須維持 channel 白名單（模組註冊時登記允許的 channel），否則破壞 contextIsolation 的隔離價值 |
| **notifyToolsChanged 回呼鏈** | `promptHandlers` 經 `onToolsChanged`（`ipcHandlers.cjs:10`）通知 MCP | repository 下沉後，notify 時機點不可遺失（create/update/delete/toggle-lock 後） |
| **rate limit / token 記憶體狀態** | `mcpServer.cjs:24` `rateLimitMap` 為 module-level | 重構 MCP 檔案結構時若搬動初始化點 → 計數歸零或洩漏 |
| **前端全量載入** | `prompts:getAll` 一次撈全部（Architecture_Discussion 已記） | repository 不改變此行為（本次不碰分頁），但要留 hook |

### 3.3 不受影響（明確劃出安全區）

- Vue 元件樹、Pinia 既有 store 的**資料形狀**（repository 回傳的 DTO 必須與現行 `prompts:getAll` 的 mapping 一致，見 `promptHandlers.cjs:71-81`）。
- auto-updater、tray、CSP、consentBroker 的**對外行為**。
- 既有 vitest 測試（`__tests__/schemaUtils.test.js`、`variables.test.js`、`PromptModal.test.js`）—— 應持續綠燈當回歸網。

## 四、實作策略與步驟（Implementation Strategy）

> 原則：**每一步都行為等價、可獨立 merge、測試持續綠燈（Strangler Fig 漸進絞殺，不大爆改）。**

### 階段 A：前置準備（解耦地基，零行為變更）

- **A1. 補測試網（FP01 §十.1 前提）**
  為 4 塊領域熱點補單元測試：變數同步、串聯展開（已有）、機密釋放策略、（未來）權益。
  目的：後續每一步都有回歸護欄。
  *依賴 memory `backend-integration-test-runtime` 的跑法。*
- **A2. 抽領域純函式**
  把 `_extractVariables / _syncAndStringifyVariables` 移到 `core/prompts/variableSync.cjs`，
  與 `src/utils/variables.js` 統一單一真相（消 C8）。handler 改 import，行為不變。
- **A3. 引入 repository（只搬 SQL，不改邏輯）**
  建 `promptRepository / folderRepository`，把 `promptHandlers / folderHandlers` 的 `db.prepare()` 搬進去。
  handler 變薄但行為等價。**先不動 MCP。**

### 階段 B：核心重構（接縫成形）

- **B1. MCP server 改用同一個 repository（消 C2）**
  把 `mcpServer.cjs:150-328` 的 inline SQL 換成 repository 呼叫；機密/串聯規則改呼叫共用領域層。
  *這是風險最高的一步，單獨一個 PR，配 E2E（`E2E/test-*.cjs`）驗證。*
- **B2. 引入 migrator + `schema_migrations` 版本表（修 C5/C6）**
  把現況 schema 定為 baseline v0；之後欄位/表改由模組宣告 migration。
  `mcp_tokens / secret_release_audit` 的 DDL 歸還其所屬模組宣告（但本階段仍可實際建在同一庫）。
- **B3. 引入 registry + moduleContext（修 C3）**
  把 prompts/folders 包成「core 模組」，用同一套契約註冊。
  `ipcHandlers.cjs` 改為「組清單 → registry」。**此時 core 與未來 add-on 寫法一致。**

### 階段 C：後續整合（權益閘門 + 前端）

- **C1. entitlement 閘門（修 C4）**
  `registry` 註冊前過 `isEntitled(module.requires)`；core 模組 `requires: null`（永遠啟用）。
  初版權益讀本地加密設定，預留授權碼驗證。
- **C2. preload 受控泛型橋 + licenseStore（修 C9/C10）**
  preload 加白名單式 `invoke`；`src/stores/licenseStore.js` 持有 entitlements 供元件 gating。
- **C3. 試煉：第一個「假 add-on」**
  以一個瑣碎付費模組（例：匯出 MD）走完整契約，驗證「沒授權→整包不掛、安全退回本地」（FP01 §三 fail-safe）。

> 之後雲端/團隊/合規屬 **greenfield**，照 FP01 §六~九，另立 PRD，不在本重構內。

### 5.2 技術阻礙與退路計畫（Rollback Plan）

| # | 技術阻礙 | 早期偵測 | 退路 / Rollback |
| :-- | :-- | :-- | :-- |
| R1 | **MCP 與 IPC 行為不一致**（B1 後 AI 取卡與 GUI 結果分歧） | E2E `test-*.cjs` + 黃金樣本比對 | repository 後該 PR 可獨立 revert；保留舊 inline SQL 於同檔註解一個 release 再刪 |
| R2 | **migrator 誤判 baseline 砸壞既有使用者 DB** | 用真實舊 DB 樣本跑升級測試 | migrator 設計為**冪等 + 失敗即不寫版本**；保留 `promptbox.db.plain.bak` 機制；提供「不動資料、只加版本表」的 dry-run |
| R3 | **preload 泛型 invoke 擴大攻擊面** | 安全 review（參考 `Doc/tool/資安審查專家.md`） | 退回「白名單 channel 才放行」；最壞回到靜態橋，模組 IPC 改 build-time 產生 |
| R4 | **stateless MCP repository 狀態污染** | 並發請求壓測 | repository 強制無狀態（只吃 db）；以 lint/review 禁 module-level 可變快取 |
| R5 | **領域層抽取漏掉機密/稽核分支** | 機密卡片 path 的整合測試（never/once/session × IPC/MCP） | 分支抽取以「兩路徑同 commit」為紀律；未完成不 merge |
| R6 | **範圍蔓延**（手滑開始蓋 server/DDD） | PR review 對照本報告 §〇 範圍 | 硬性：server 側另開 PRD，本重構 PR 一律拒收 server code |
| R7 | **Native module（better-sqlite3-multiple-ciphers）重 rebuild** | 每階段跑一次 `electron:build`（Architecture_Discussion 已記風險） | 鎖 Electron 版本；migrator/repository 不碰 native 介面，降低牽連 |

### 5.3 Go / No-Go 建議

- **強烈建議做**：A 全部 + B1 + B2（消滅 C2 雙實作、建立 migration 版本控管）—— 這兩塊**即使最後不賣插件也是淨賺**（DRY + 可測 + DB 可維護）。
- **看商業確定性再做**：B3 + C 全部（registry / entitlement / preload 泛型橋）—— 這些只有在「確定要出付費模組」時才回本，否則是 YAGNI。
- **明確不做（現在）**：server 側 DDD、雲端、合規控制項 —— 等第一個付費模組收到錢、契約被真實驗證後再說（呼應 ADR-002 §6「不平行開發」）。
