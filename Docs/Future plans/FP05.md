## 二、需求本質與已定的 5 個決策

**真正驅動力**：目前主要是「**嫌亂 + 為未來留權限隔離的餘地**」。目前 **Private + 單人、無協作者**；「不同人存取」是**假想未來**。

| # | 決策 | 定案 |
| :-- | :-- | :-- |
| 1 | Doc 去向 | **搬新 repo，且開成 private**（權限是 GitHub repo 層級控制，資料夾位置擋不住人） |
| 2 | App 檔案位置 | **升到 repo 根**（連帶改 CI `working-directory: ./PromptBox` → `.`） |
| 3 | 歷史 | **不保留**，且**希望 Doc 歷史從 app repo 消失** |
| 4 | 根層雜項歸屬 | `E2E` / `README.md` / `下次跟新.text` → **App**；`L0` / `L1` / `TESTMCP` / `img` → **Doc** |
| 5 | 並存方式 | **multi-root workspace**（一個 `.code-workspace`，同視窗掛兩個 folder，見 §五） |

## 三、關鍵判斷：private + 單人 → 風險歸零

上一輪原本擔心的東西，因為「private + 單人」全部消失：

- **沒有外洩問題**：repo 一直是 private，Doc 從未公開；「歷史抹不乾淨、fork/cache 攔不住」皆不適用。
- **force-push 100% 安全**：只有一個人，沒有別人的 clone 會被打爛。
- **第 2 點（升到根）＋第 3 點（Doc 歷史消失）可一次完成**：重建 app repo 時天生就不含 Doc。

> **老實提醒（划算度）**：拆兩個 repo 本質是為「**權限隔離**」而生的重工具。若純粹嫌亂，其實「單一 repo 內把根層收進子資料夾」就能解掉八成。**拆 repo 的正當理由是未來的 code/docs 權限隔離**——若那個未來夠真，趁單人乾淨時拆是對的時機（有協作者後再拆，歷史與習慣遷移更麻煩）。

## 四、做法（下次操作照這個走）

目標磁碟結構：

```
workspace/                       ← 純容器，不是 git repo
  PromptBox/                     ← repo A → tsa206531/PromptBox（app，內容提到根）
  Doc/                           ← repo B → tsa206531/PromptBox-Doc（private，含 L0/L1/TESTMCP/img）
  PromptBox.code-workspace       ← multi-root workspace（見 §五）
```

### Step 0 — 先備份

```bash
# 整包複製一份，任何步驟出錯都能回復
cp -r "j:/desktopp/githut/PromptBox" "j:/desktopp/githut/PromptBox_BACKUP_20260706"
```

### Step A — 建 Doc private repo（全新、不帶歷史）

1. GitHub 上建 **private** repo，例如 `tsa206531/PromptBox-Doc`。
2. 準備 `workspace/Doc/`，內含：原 `Doc/*` ＋ 搬進來的 `L0/` `L1/` `TESTMCP/` `img/`。
3. 初始化並推送：

```bash
cd workspace/Doc
git init
git add .
git commit -m "docs: initial import (split from monorepo)"
git branch -M main
git remote add origin https://github.com/tsa206531/PromptBox-Doc.git
git push -u origin main
```

### Step B — 重建 App repo（續用原 remote、內容提到根、丟棄含 Doc 的舊歷史）

因為決策 3「不保留歷史」，**最簡最乾淨的是 fresh init**（不必用 `git filter-repo`）：

1. 準備 `workspace/PromptBox/`，內容 = 原 `PromptBox/*` **提升到這層的根**，再把根層的 `E2E/` `README.md` `下次跟新.text` `.github/` 一起放進來。
2. 重建：

```bash
cd workspace/PromptBox
rm -rf .git                       # 拋棄舊的混合歷史（含 Doc 的 9 個 commit）
git init
git add .
git commit -m "chore: repo split — app only, promoted to root, docs removed"
git branch -M main
git remote add origin https://github.com/tsa206531/PromptBox.git
git push -u origin main --force   # 覆蓋 GitHub 上舊的混合歷史
```

3. **修 CI**：把 `.github/workflows/release.yml` 的 `working-directory: ./PromptBox` 改成 `.`（app 已在根）。

> **替代方案（若哪天反悔想留 code 歷史）**：改用 `git filter-repo --path PromptBox/ --path E2E/ --path README.md --path 下次跟新.text --path .github/ --path-rename PromptBox/:`（保留 commit 歷史、剔除 Doc、把 `PromptBox/` 內容提到根），再 force-push。本案因決策 3 選 fresh init。

### Step C — 建 `.code-workspace`（見 §五）並清乾淨

- 在 `workspace/` 放一份 `PromptBox.code-workspace`（建議**不提交**進任一 repo，或只留本機）。
- 確認 `L0/L1/TESTMCP/img` 已在 Doc repo、不再殘留於 app repo。

## 五、multi-root workspace 是什麼

VSCode 一個視窗預設只開一個資料夾；但可用一個 `.code-workspace` 檔案在**同一個視窗掛多個彼此獨立的資料夾**（稱為 workspace 的「根 / roots」，彼此是 sibling）。

`PromptBox.code-workspace` 內容大致：

```json
{
  "folders": [
    { "path": "PromptBox" },
    { "path": "Doc" }
  ]
}
```

雙擊它 → VSCode **一個視窗、檔案總管同時列出 PromptBox 與 Doc 兩棵樹**，Source Control 面板**分別顯示兩個 repo**，各自 commit、各自 push 到不同 GitHub。

**對本案的意義**：日常仍「一個視窗看到全部」，寫 code 順手翻文件的體驗不變；但底層是兩個權限分離的獨立 repo——未來給只碰 code 的人 clone app repo 時，**他們根本拿不到、也看不到 Doc**，正好命中「docs 不給別人看」。

## 六、要小心的地雷

### 6.1 別動到 release tags / auto-update（最重要）

- app 有 `v3.2.1` 等 tag，已安裝的 app 靠 `tsa206531/PromptBox` 的 **Releases** 抓更新。
- **GitHub Releases 的二進位檔掛在 tag 名稱上、是 server 端物件，force-push 重寫歷史不會刪掉它們。**
- **只要 repo 名稱與 `publish` 設定（`PromptBox/package.json` owner/repo）維持不變，auto-update 就不受影響。**
- 真正做時**驗一下**：Releases 頁面資產仍在、`dev-app-update.yml` 與 `publish` 未被改動。

### 6.2 Claude 記憶路徑

- `.claude/` 在根層且已 gitignore；記憶目錄 key 由工作區路徑 `j--desktopp-githut-PromptBox` 推出。
- **若磁碟路徑改變**（例如 app 移到 `workspace/PromptBox`），Claude 專案 key 會變，現有記憶不會自動載入 → 需注意遷移。

### 6.3 `.vscode/` 與根層設定

- 決定原根層 `.vscode/`、`.gitignore`、`node_modules`（npx/vitest 殘留）搬去哪，或改放進 `.code-workspace`。

## 七、要接受的 trade-off：失去原子性

拆完後，**改一個功能 ＋ 它的 PRD/TD 會變成兩個 repo、兩次 commit、兩個 PR**，沒法綁在一起。本專案版本號與 Doc 綁很緊（PRD/TD/Wishlist 對應版本號），單人作業時這個來回切換會有點煩。**這是拆分的固定成本，要先接受。**
