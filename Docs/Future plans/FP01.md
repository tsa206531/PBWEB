## 〇、一句話結論

> 別問「Clean Architecture 還是 DDD」，問「我的**模組契約**長怎樣、**權益閘門**擺哪裡」。
>
> - **六角架構（Hexagonal / 埠與配接器）**：從 add-on 一路適用到雲端／團隊／企業／醫療，投資複利。
> - **DDD**：桌面 core 只點在 4 塊規則熱點；到**團隊版的 server 側**才全面回本。
> - 你列的「版本」要先拆成兩類：**部署模式（mode）** 與 **合規規範（compliance regime）**——後者不是「架構」出來的版本。

---

## 一、現況診斷：Transaction Script，不是分層架構

目前的真實分層是 **交易腳本（Transaction Script）** 模式：

```
Vue 元件 → Pinia store → preload → IPC handler ──┐
                                                  ├─ 同一個函式裡做完：
                                          參數驗證 + 業務規則 + SQL + 加密
```

證據：`electron/handlers/promptHandlers.cjs` 的 `prompts:create` 一個 handler 內同時做了
解析 folder、跑變數同步規則 `_syncAndStringifyVariables`、組 SQL、寫 DB、發 MCP 通知。
**傳輸層、業務層、資料層焊死在一起。**

- 對 100 筆 prompt 的單機工具，這寫法**又快又好懂**，沒有抽象稅——不是缺點。
- 唯一問題：**焊死的東西沒辦法當可插拔插件抽走。** AI 包、團隊包要當 add-on，
  就不能讓邏輯長在 core 的 handler 裡。
- 目前唯一稱得上「層」的是 `services/`（cryptoService / keyService / consentBroker），
  方向對，是接縫雛形。

## 二、為什麼不全套 Clean Architecture + DDD

| 維度 | DDD 適合的場景 | PromptBox 實況 |
| :-- | :-- | :-- |
| 領域複雜度 | 業務規則纏繞、語意衝突多 | 8 成是 CRUD，只有 4 處有真規則 |
| 團隊規模 | 多人、需統一語言 | 接近單人／小團隊 |
| 變更頻率 | 規則常變、需隔離 | 規則相對穩定 |
| 成本 | Aggregate/VO/Repository/領域事件的儀式 | 會拖慢出第一個付費版本 |

DDD 的回報來自「馴服複雜領域」，但桌面 core 的領域不夠複雜到需要被馴服。
硬上全套，會花三個月把 CRUD 包成 Entity/Repository/UseCase，換來的可插拔性用更輕的手段就能拿到。

**推薦：Hexagonal-lite（埠與配接器精簡版）+ 在刀口上用 DDD。**
只取 Clean Architecture 裡「製造接縫」那塊（依賴反轉 + 薄 use-case 層 + repository 抽象），
丟掉 DDD 的儀式。

## 三、真正的關鍵：兩個接縫

賣插件，技術上只需要兩樣東西。

### ① 模組契約（Module Contract）

每個功能模組（含付費 add-on）是自我描述的單元，向 core 註冊：

```
一個 module = {
  ipcChannels:  要掛哪些 IPC channel（命名空間隔離，如 ai:*, team:*）
  migrations:   自己擁有的 DB 表/欄位（自帶版本）
  mcpTools:     要對外曝露的 MCP tool
  requires:     啟用所需的權益 key（如 'pro', 'team'）
  register:     一個 register(ctx) 函式，core 啟動時呼叫
}
```

core 啟動時跑 registry，**逐一問每個模組「你被授權嗎？被授權才註冊」**。
沒授權 → 整包不掛，安全退回純本地（= ADR-002 §1 的 fail-safe 落成程式碼）。

### ② 權益閘門（Entitlement Gate）

一個地方判定「這台安裝能啟用哪些模組」，所有模組註冊前都過這關（= ADR-002 §4）。

### 目標目錄結構

```
core/                    ← 永遠存在、純本地、可驗證的乾淨核心
  modules/
    prompts/             ← 連 core 功能都用同一套契約寫
    folders/
modules/                 ← 可選 / 付費 add-on，同一個契約
  ai/        (requires: pro)
  team/      (requires: team)
platform/
  registry.cjs           ← 註冊中心
  entitlement.cjs        ← 權益閘門
```

> **重點**：連 core 的 prompts/folders 都用同一套契約寫，避免「core 一種寫法、add-on 另一種寫法」的雙軌技術債。

## 四、插件版本的三種做法（選型）

| 做法 | 隱私可驗證性 | 防破解 | QA 成本 | 評語 |
| :-- | :-- | :-- | :-- | :-- |
| **A. 編譯期乾淨 build**（付費碼不在 binary 裡） | 🟢 最強 | 🟢 強 | 🔴 build 矩陣變多 | **隱私版／企業地端版必走** |
| **B. Runtime 權益旗標**（程式都在，授權碼解鎖） | 🔴 弱 | 🟡 易被 patch | 🟢 單一 build | 跨模式功能（AI 包）可用 |
| **C. Runtime 動態載入第三方插件** | 🔴 弱 | 🔴 攻擊面大 | 🔴 災難 | **早期千萬別碰** |

**推薦組合：**
- 隱私版 / 企業地端版 → **A**：賣點「聯網程式碼根本不在裡面」，要可被驗證。
- Pro 的 AI 包、進階變數等跨模式功能 → **B**：本來就需聯網，用旗標最省事。
- **C 不要做**：runtime 第三方插件 = 授權、安全、QA 三重地獄。要的是「**第一方模組，在 build／授權時選擇**」，不是開放外掛市集。

A 與 B 不衝突——共用第三節的模組契約，差別只在「模組被編譯排除，還是被權益關掉」。

## 五、DDD 用在刀口上：桌面 core 只抽這 4 塊

抽成獨立純函式領域層（無 IO、好測試）的理由是**為了能單元測試**，不是為了 DDD：

1. **變數同步**（`_syncAndStringifyVariables`，現埋在 handler 裡）
2. **卡片串聯展開**（`expandChains` / `schemaUtils`，已半獨立，做得好）
3. **機密釋放策略**（never／once／session + consent 流程）
4. **權益判定**（未來核心規則）

> 對應 ADR-002 自陳的最大風險：**多版本 × 零自動化測試 = 回歸風險暴增。**
> 架構乾不乾淨，比不上測試覆蓋率重要。重構只該做到「能開出 entitlement 接縫」與「能被測試」為止，別為純粹而重構。

## 六、雲端／團隊／企業／醫療：先拆成兩種生物

使用者提出的四個方向，**不是同一種東西**：

| 項目 | 本質 | ADR-002 用語 |
| :-- | :-- | :-- |
| 1. 雲端同步 | **部署拓樸改變**（跨出 client/server 邊界） | **mode（模式）** |
| 2. 團隊共同 | **部署拓樸改變**（多人、多租戶、RBAC） | **mode（模式）** |
| 3. ISO 27001 / SOC2 | **合規規範**（控制項 + 流程 + 證據） | 不是 edition |
| 4. 醫療級 | **合規規範 + 可能是受監管裝置** | 不是 edition |

> **你不會「開發一個 SOC2 版本」，你是「營運一個通過 SOC2 稽核的服務」。**
> **你不會「fork 一份醫療 codebase」，你是「在某個 mode 上疊合規控制項 + 跑一套受監管的開發流程」。**
> 把 3、4 當成「第四、第五個 edition 去開發」，是最容易燒掉小團隊的誤會。

### 六角架構：適用且回本加倍

為 add-on 鑿的埠，到 client/server 拆分時原地升級成網路邊界。
prompts repository 介面：本地版接 SQLite 配接器、雲端版接 HTTP 配接器，**領域邏輯一行不改**。
前一步的接縫投資在此複利。

### DDD：團隊版 server 側才全面回本

團隊版引入真正纏繞的領域（租戶、成員、角色、權限、共享庫、衝突、稽核、保存策略），
有真 invariant（「Viewer 不能改別人卡片」「稽核日誌 append-only 不可竄改」）與真・統一語言。
此時 server 側值得**真的劃 bounded context**：

```
Identity & Access context   ← 登入、RBAC、租戶
Prompt context              ← 現有核心領域
Sync context                ← 備份/還原/衝突
Audit & Compliance context  ← 稽核、保存、釋放策略
```

但**桌面 core 維持 Transaction Script 不變**。一個產品同時存在「貧血 CRUD core」與「DDD server domain」並不矛盾。

## 七、逐項過一遍（本質 + 架構衝擊）

**1. 雲端同步** — ADR-002 §3.3 已釐清：不是 real-time 雙向同步，是「帶 Token 的自動備份/還原」。
架構衝擊小（一個 syncService + Storage 配接器），刻意避開 conflict resolution 無底洞。**務實，先做。**

**2. 團隊共同** — **真正的大跳躍。** 多人 = 多租戶 + RBAC + 並發 + 衝突 + 稽核，
你開始營運後端服務（on-call、安全面、SaaS 維運），與隱私品牌直接對撞（ADR-002 §5 核心張力）。DDD 從這裡回本。

**3. ISO 27001 / SOC2** — **與程式分層幾乎無關。**
- ISO 27001 = 你「組織」的資安管理體系（ISMS）認證，風險導向控制項。
- SOC2 = 對你「營運的服務」的稽核報告（Type II 要 6–12 個月證據累積）。
- 關鍵：**純本地桌面 App 幾乎沒東西可稽核**；ISO/SOC2 只在你跑了雲端／團隊後端後才有意義，
  **預設了第 1、2 項存在**。架構能幫的只是把稽核日誌、加密、存取控制的接縫留好，剩 80% 是流程與證據。

**4. 醫療級** — 最深的水，先分清一個分岔：
- **只是「存的 prompt 剛好用於醫療」且碰到 PHI** → 多半落在 HIPAA／當地個資法 + BAA，
  重點是加密、存取控制、稽核、資料在地化。重，但可控。
- **軟體參與臨床決策 = 受監管醫療裝置（SaMD）** → 觸發 IEC 62304（軟體生命週期）、
  ISO 14971（風險管理）、ISO 13485（品質系統），甚至 FDA／MDR 送審。
  **此時架構只佔總工作量 ~20%，剩下是 QMS、可追溯性、風險檔案、驗證文件。**

> ⚠️ 待澄清：醫療版是「HIPAA 等級的工程任務」還是「受監管裝置等級的公司級承諾」——這個分岔決定一切，先別急著選。

## 八、合規版本「怎麼做」：三件事疊上去

不是寫一個版本，是疊三層：

1. **控制項（Controls）**：加密（已有 SQLCipher）、RBAC、append-only 稽核
   （`secret_release_audit` 已是雛形）、資料保存/刪除策略、存取記錄。← 這些是 code，靠 Audit/Access context 提供接縫。
2. **流程（Process）**：變更管理、存取審核、事故應變、員工權限回收……是公司運作，不是 git repo。
3. **證據（Evidence）**：持續產生可被稽核員查的紀錄。

> 架構的任務只是把 1 的接縫留乾淨，讓 2、3 有東西可掛。別想著「把 SOC2 寫進程式」。

---

## 九、戰略排序與隱私品牌

依賴關係：**3、4 蓋在 1、2 上**；1、2 互相獨立。天然順序：

```
雲端同步(1) ──┐
              ├─→ 做成「能 cloud 也能 on-prem 同一份 code」（六角配接器互換）
團隊(2) ──────┘        │
                       ├─→ 疊 ISO/SOC2 控制項(3)
                       └─→ 疊 HIPAA/醫療控制項(4)
```

關鍵洞見（呼應 ADR-002 §5）：**企業/醫療客戶最在意「資料別離開我機房」，與隱私品牌同向。**
聰明做法——把團隊領域寫成 **deployment-agnostic**：同一份 server code，配接器一換就能部署成
雲端 SaaS **或** 客戶地端/air-gapped。於是醫療版不是另一條 codebase，是
「團隊版 + 地端部署 + 醫療控制項 + 一套 IEC 62304 流程」。

> **六角架構讓這件事可能；DDD 讓那個 server 領域不會在合規壓力下崩成義大利麵。** 這是它倆在這四項的真正價值。

---

## 十、務實下一步（呼應 ADR-002 §6，點破前提）

1. **先補測試地基**（尤其第五節 4 塊純邏輯）——沒有它，多 edition 是裸奔。
   （後端整合測試跑法見 memory：`backend-integration-test-runtime`。）
2. **把 prompts/folders 重寫成模組契約格式**（一次一個，行為不變，練契約）。
3. **長出 registry + entitlement 閘門**（薄薄一層即可）。
4. **第一個付費模組走通**（建議 AI 包走 B 旗標，最快收到錢驗證契約）。
5. 之後每個 mode／edition 照契約逐一填，**不平行開發**。
